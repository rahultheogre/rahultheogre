Detection
Pipeline

A dual-tier fraud detection system combining traditional graph ML (CPU) with GraphSAGE-based deep graph learning (GPU). Detects organized fraud rings, individual bad actors, and novel fraud patterns using a multi-layer graph architecture over 186 engineered features.

Graph ML GraphSAGE Leiden Detection XGBoost SHAP Snowflake NetworkX Dual-Tier Architecture
Problem

Insurance fraud rings don't show up in individual records.

Traditional ML models score members in isolation — they miss organized fraud where individual behaviour looks normal but the network pattern is anomalous. A member who shares a phone number with 12 others, all making high-cost claims in the same zip code, looks clean in row-level feature space.

The system needed to detect both individual fraud signals and organized ring structures simultaneously — across roadside assistance claims, cost patterns, contact sharing, and timing coordination.

Architecture

Dual-Tier System

Two tiers deployed based on computational budget and detection requirement. Tier 1 runs in production for real-time scoring. Tier 2 runs periodically for deep ring analysis.

Tier 1 — CPU · Real-Time

Leiden + PageRank + XGBoost

Cost-effective, interpretable, production-ready. Graph metrics augment 186 engineered features for XGBoost classification.

  • 01 → Graph construction from relationship tables
  • 02 → Leiden community detection
  • 03 → PageRank + centrality computation
  • 04 → Graph features merged with 186 base features
  • 05 → XGBoost fraud probability scoring
  • 06 → SHAP explanation for investigators
Tier 2 — GPU · Batch Analysis

GraphSAGE Unsupervised + Supervised

Deep graph learning. Discovers novel fraud patterns without labeled examples. Inductive — scores new members immediately.

  • 01 → Multi-layer graph construction
  • 02 → Leiden + PageRank (enhanced multi-resolution)
  • 03 → GraphSAGE unsupervised — 128-256 dim embeddings
  • 04 → Feature fusion: embeddings + 186 features + graph
  • 05 → GraphSAGE supervised — fraud probability
  • 06 → Role detection: organizer / participant / mule
Graph Layers

Multi-Layer Graph Architecture

LAYER 01

Member

membership_number as primary nodes. Membership level, type, tenure, status as node attributes.

LAYER 02

Contact

Phone, email, address as connecting nodes. Shared contacts create edges between members — primary fraud ring signal.

LAYER 03

Service

Call events, costs, timing as relationship weights. Coordinated call patterns and cost anomalies encoded as edge weights.

LAYER 04

Geographic

Zip codes, address clusters as spatial nodes. Geographic proximity combined with usage correlation reveals local fraud operations.

Design Decisions

Why these choices, not others.

WHY A DUAL-TIER ARCHITECTURE?

Real-time scoring and deep ring analysis have fundamentally different compute profiles

GPU-based GraphSAGE is powerful but takes minutes to run over large graphs — not suitable for real-time claim scoring. XGBoost with graph-derived features runs in milliseconds on CPU and is fully interpretable for investigators. The dual-tier approach gives you fast, explainable production scoring (Tier 1) plus periodic deep analysis that catches sophisticated rings Tier 1 misses (Tier 2). Each tier informs the other — Tier 2 ring profiles update Tier 1 community features.

// Rejected: Single GPU model for all scoring (latency too high for real-time), pure rule-based (misses novel patterns)
WHY LEIDEN OVER LOUVAIN FOR COMMUNITY DETECTION?

Leiden guarantees well-connected communities — Louvain can produce internally disconnected ones

Louvain is fast but has a known flaw: it can assign nodes to communities they're not well-connected to, producing structurally invalid clusters. For fraud ring detection, a disconnected community is a false positive. Leiden fixes this by refining the partition after each merge step, guaranteeing that every community is internally connected. This matters when you're using community membership as an investigative signal — investigators need to trust that community members are actually linked.

// Rejected: Louvain (disconnected community problem), label propagation (unstable, non-deterministic)
WHY GRAPHSAGE OVER A STANDARD GNN?

GraphSAGE is inductive — it generalizes to new members without retraining

Standard transductive GNNs (GCN, GAT) learn embeddings for fixed nodes in a fixed graph. When a new member joins, you have to retrain the entire model. GraphSAGE learns an aggregation function over sampled neighbourhoods instead — so it can immediately generate an embedding for a new node by aggregating its neighbours' features. For fraud detection where new accounts are created constantly, this is critical.

// Rejected: GCN (transductive, no new-member inference), DeepWalk (ignores node features)
WHY UNSUPERVISED GRAPHSAGE BEFORE SUPERVISED?

Labeled fraud data is scarce — unsupervised pre-training on graph structure compensates

Confirmed fraud labels are expensive and lag real behavior. Unsupervised GraphSAGE learns rich structural embeddings from the full graph without labels — capturing network position, community role, and relationship patterns. These embeddings are then used as input to the supervised classifier, which only needs labels for the final prediction layer. This dramatically reduces the labeled data requirement and means the model still learns from the majority of unlabeled members.

WHY AUGMENT XGBOOST WITH GRAPH FEATURES RATHER THAN REPLACE IT?

XGBoost + graph features outperforms either alone — and stays interpretable

Graph centrality metrics (PageRank, betweenness, community fraud ratio) capture network-level signals that tabular features miss entirely. But XGBoost on tabular data alone already captures strong individual behavioral signals (call margin, cost anomalies, contact sharing counts). Combining them gives the model both perspectives. Crucially, XGBoost with SHAP remains fully interpretable — investigators can see exactly which features drove a fraud flag, which is a compliance requirement in insurance.

// Rejected: Pure graph model (loses tabular feature signal, less interpretable), pure tabular XGBoost (blind to ring structures)
WHY WEIGHTED RELATIONSHIP EDGES (2.5–3.0 / 1.0–2.0 / 0.3–0.8)?

Not all shared contacts carry equal fraud signal — weight encodes domain knowledge

Exact phone sharing between two adults in different households is a strong fraud indicator (weight 2.5–3.0). Similar usage timing patterns are weaker — many legitimate members call at similar times (weight 0.3–0.8). Encoding these weights into the graph means community detection and centrality algorithms naturally prioritize strong-signal edges when identifying rings, rather than treating all relationships as equivalent. Weights were derived from domain expert input and validated against confirmed fraud cases.
