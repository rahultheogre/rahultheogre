Platform

A voice-first multi-agent sales platform built on LangGraph — a supervisor routing live calls across 5 specialised agents, grounded in a Neo4j knowledge graph and Microsoft GraphRAG, with DSPy-optimised prompts for production reliability.

LangGraph Multi-agent Neo4j GraphRAG Leiden Community Detection DSPy Twilio Deepgram STT LiveKit WebRTC
Problem

A single LLM cannot handle the full scope of a live sales call — and mistakes in voice are unrecoverable.

Sales calls span multiple distinct tasks: qualifying SMB leads, handling general support questions, booking appointments, managing CRM data, and routing to human agents. A monolithic LLM handling all of these either gets overwhelmed by competing instructions or defaults to a generic tone that kills conversion. What was needed was specialist behaviour per intent — with a supervisor that could route mid-conversation without the caller noticing the handoff.

Voice compounded the reliability requirement. A hallucinated answer in a chat interface can be edited. On a live call it damages trust immediately. Every agent response needed to be structurally constrained and grounded in the company's actual sales rules and conversation policies — not the model's priors.

Design Decisions

Why these choices, not others.

WHY LANGGRAPH OVER SIMPLE CHAINS OR AUTOGEN?

Sales calls are stateful, conditional, and mid-call routing is non-linear — chains can't model that

Simple LangChain chains execute a fixed sequence. A sales call doesn't follow a fixed sequence: a support question mid-pitch needs an immediate agent swap without losing conversation state, then a return to the sales agent. LangGraph models this as an explicit state machine — nodes are agents, edges are conditional routing rules, and the full conversation state is maintained and passed across every hop. AutoGen was evaluated but its agent communication model is too loosely coupled for the strict handoff semantics a live call requires.

// Rejected: LangChain sequential chain (no conditional mid-conversation routing), AutoGen (loose coupling, harder to enforce structured outputs per agent)
WHY NEO4J OVER A VECTOR STORE?

Sales knowledge is relational — a vector store cannot express "this rule applies only to users in segment X with policy Y"

A pure vector store retrieves semantically similar text. But the sales knowledge here was conditional and relational: a conversation policy might apply only to SMB leads above a certain deal size, linked to specific product rules, which are further linked to user segment profiles built from interaction history. These multi-hop relationships cannot be expressed as proximity in embedding space. Neo4j lets agents traverse the exact edges that matter for a given caller's context — not just find similar text.

// Rejected: Qdrant vector store (no relationship traversal, can't encode conditional rule chains), hybrid vector+SQL (two systems to maintain, joins not graph-native)
WHY GRAPHRAG WITH LEIDEN COMMUNITY DETECTION?

Individual node retrieval misses entity clusters — Leiden finds the community of related concepts agents need

Microsoft GraphRAG augments graph retrieval with community-level summaries built using the Leiden algorithm. Leiden detects densely connected subgraphs in the knowledge graph — for example, a cluster of sales rules, the product types they govern, and the user segments they target. Instead of retrieving individual nodes, agents receive a pre-summarised community context covering all relevant interconnected entities. This is particularly valuable for the Sales/SMB agent, where answering a pitch question correctly requires simultaneously knowing the product, the rule set, and the caller's history.

// Rejected: plain Neo4j Cypher retrieval only (misses emergent entity clusters), vanilla RAG over graph text exports (loses structural relationships)
WHY DSPY FOR PROMPT OPTIMISATION?

Hand-written prompts for 5 agents across a changing knowledge graph are brittle — DSPy compiles reliable prompts from examples

Each of the 5 agents needed prompts grounded in the company's specific conversation policies and sales rules. Writing and maintaining these by hand as the knowledge graph evolved was unsustainable. DSPy's bootstrap few-shot compiler takes labelled input/output examples and optimises prompt structure automatically, including chain-of-thought reasoning steps. This gave each agent a stable, policy-grounded prompt without prompt engineering becoming a full-time job — and made reliability testable rather than subjective.

// Rejected: manual prompt engineering (brittle under knowledge graph updates), zero-shot prompting (unacceptable hallucination rate on policy-specific questions)
WHY DEEPGRAM STT OVER WHISPER?

Streaming real-time transcription — Whisper's offline batch model adds unacceptable latency for a live call

OpenAI Whisper is a strong transcription model but it processes audio in completed segments, not streams. For a voice sales platform, the agent needs to begin formulating a response while the caller is still speaking — streaming transcription is non-negotiable. Deepgram's API provides word-level streaming with low-latency partial transcripts, which lets the LangGraph supervisor begin intent classification before the caller finishes their sentence, shaving perceptible response delay from the conversation.

// Rejected: OpenAI Whisper (batch processing, no streaming partial transcripts), Google Speech-to-Text (higher cost at volume, latency comparable)
