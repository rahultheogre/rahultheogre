Problem

Corporate trainers were spending more time finding answers than delivering training.

Netcom's training team worked from a large, fragmented corpus of PDF documentation — certification guides, technical manuals, policy documents, past Q&A transcripts. Preparing for a session meant manually searching across multiple PDFs, reconciling conflicting versions, and constructing answers from scratch. Two hours of prep per session was the norm.

The deeper problem was retrieval quality. Standard keyword search returned the wrong section of the right document, or the right keyword from a section that predated a policy change. Trainers needed a system that understood the question, not just its tokens — and that could surface the most relevant passage from hundreds of pages in seconds, with a generated answer they could verify against the source.

Architecture

Adaptive Retrieval Pipeline with 3-tier Cache

STAGE 01

PDF Ingestion & Adaptive Chunking

Complexity-driven chunk sizing: 256 tokens for dense technical content, up to 1024 for narrative sections. Overlap preserved for boundary coherence. Nomic AI embeddings generated per chunk.

STAGE 02

Hybrid Search

Qdrant vector store for dense semantic retrieval + BM25 keyword retrieval in parallel. Results merged — semantic catches paraphrase, keyword catches exact terminology and acronyms.

STAGE 03

Cross-encoder Reranking

Top candidates from hybrid search scored jointly against the query. Cross-encoder reads query + chunk together — captures relevance nuances the bi-encoder misses at retrieval time.

87% retrieval accuracy
STAGE 04

GPT Response + Cache

Top reranked chunks passed to GPT with source attribution. 3-tier cache: Redis (hot queries), in-memory LRU (session), file-based (cold). Repeated queries skip all stages.

80% faster on repeat queries

Design Decisions

Why these choices, not others.

WHY ADAPTIVE CHUNKING OVER FIXED-SIZE?

Chunk size should match content density — not an arbitrary token count

Fixed 512-token chunking splits dense technical tables mid-row and leaves narrative sections under-contextualised. We scored each document section for lexical density and structural complexity, then sized chunks accordingly: 256 tokens for specification-heavy content where every sentence is load-bearing, up to 1024 for explanatory prose where context requires more window. This kept retrieved chunks semantically complete — the reranker received whole units of meaning, not truncated fragments.

// Rejected: fixed 512-token chunking (breaks tables, under-contextualises prose), sentence-level chunking (too granular for technical docs, retrieval noise)
WHY HYBRID SEARCH (SEMANTIC + KEYWORD)?

Dense retrieval misses exact terminology; keyword search misses paraphrase — the corpus needs both

Training documentation is full of product names, certification codes, and acronyms that trainers query verbatim. A pure dense retrieval system with Nomic embeddings would return semantically similar text but potentially miss the exact section containing "CCNA 200-301 objective 3.4" because the embedding space smooths over specific codes. BM25 anchors exact-match retrieval. The hybrid merge means a query phrased naturally and a query using the precise code both land correctly.

// Rejected: pure dense retrieval (misses exact codes and acronyms), pure BM25 (no paraphrase, synonym, or intent matching)
WHY NOMIC AI EMBEDDINGS OVER OPENAI ADA-002?

60% cost reduction with comparable retrieval quality on technical documentation

The corpus required re-embedding on document updates. At OpenAI ada-002 pricing, frequent re-ingestion of the full PDF corpus was expensive. Nomic Embed is open-weights, deployable on-prem, and produced comparable retrieval quality on this domain after benchmarking against the same query set. For a system where embedding happens at ingestion (not on every query), the per-token cost of the embedding model directly affects total operational cost.

// Rejected: OpenAI ada-002 (60% higher embedding cost at this corpus volume), BGE-M3 (slower, heavier for deployment constraints)
WHY 3-TIER CACHE INSTEAD OF JUST REDIS?

Different query patterns have different latency budgets and infrastructure costs

Redis handles genuinely hot queries across sessions — popular certification topics trainers ask every cohort. In-memory LRU handles within-session repetition, which is common as trainers refine their phrasing of the same question. File-based cache handles queries that are cold to Redis but not worth a full pipeline run — occasional queries that recur weekly. Collapsing to Redis alone would increase Redis memory footprint and miss the within-session hits that are fastest to serve from process memory.

// Rejected: Redis-only cache (misses in-session repetition, higher memory cost), no cache (GPT latency on every query is user-hostile for a training tool)
WHY CROSS-ENCODER RERANKING AFTER HYBRID SEARCH?

Bi-encoders embed query and document independently — they cannot model their interaction

The Nomic bi-encoder produces separate embeddings for query and chunk, then compares them by cosine similarity. This is fast but loses the fine-grained relevance signal that comes from reading them together. A cross-encoder takes the query and each candidate chunk as a single input sequence and scores relevance jointly — it sees exactly how the query's terms relate to the chunk's structure. Running it over the full corpus is prohibitive, but running it over the top-20 hybrid results is cheap and meaningfully improves final answer quality.

// Rejected: no reranking (bi-encoder top-1 frequently wrong on ambiguous queries), cross-encoder over full corpus (inference cost prohibitive at scale)
