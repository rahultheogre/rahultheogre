Optimizer

A production-grade planogram optimization engine using Simulated Annealing. Balances 7 competing business objectives simultaneously — sales, adjacency, utilization, movement cost, inventory gaps, shelf appearance, and constraint violations.

Problem

Retail planogram optimization is a combinatorial nightmare.

Product placement on retail shelves affects sales directly — but optimizing it means balancing spatial constraints, product family groupings, sales performance, shelf utilization, and store-associate movement cost all at once. There is no closed-form solution.

Manual optimization by category managers is slow, subjective, and doesn't scale across hundreds of planograms. The system needed to explore a massive solution space and find near-optimal placements automatically.

Architecture

Layered Optimization Engine

LAYER 01

Preprocessing

Load planogram state, fetch performance data via API, build product proximity matrix from ML feature importances. Categorize products by priority: new adds, adjusted, stable.

LAYER 02

Annealing Loop

Simulated Annealing with Metropolis criterion. T=1000, cooling rate 0.95. Neighbor generation selects product + operation based on temperature and priority tier.

LAYER 03

Multi-Objective Eval

7 weighted objectives scored per candidate solution. Hard constraints rejected outright. Soft objectives aggregated into a single energy value to minimize.

LAYER 04

Post-Processing

Audit report generation: position-level diffs, fixture utilization summary, execution task breakdown. Persisted to database via API.

// Simulated Annealing core loop T = 1000.0, cooling_rate = 0.95, min_T = 0.01 while T > min_T: neighbor = get_neighbor(current, T) // priority-weighted selection ΔE = evaluate(neighbor) - evaluate(current) if ΔE < 0: accept // improvement else: accept with P = exp(-ΔE / T) // escape local optima T = T * cooling_rate // 7 objectives weighted and summed → energy to minimize energy = Σ ( sales·0.4 + adjacency·0.2 + utilization·0.15 + movement·0.1 + inventory_gap·0.1 + empty_shelves·0.05 )
Design Decisions

Why these choices, not others.

WHY SIMULATED ANNEALING?

The solution space is too large and discontinuous for gradient-based methods

Planogram optimization is a combinatorial problem — discrete product positions, hard spatial constraints, non-differentiable objectives. Gradient descent doesn't apply. SA is well-suited because it naturally handles discrete operations (move, swap, adjust facing) and its temperature schedule gives principled control over exploration vs exploitation. It also produces interpretable state transitions at every step.

// Rejected: Genetic algorithms (slower convergence for this problem size), ILP solvers (don't scale to 100+ products with spatial constraints), greedy local search (gets trapped in local optima)
WHY SEPARATE HARD AND SOFT CONSTRAINTS?

Mixing constraint violations into the objective function creates unstable energy landscapes

If spatial violations (overlaps, width exceeded) are just another weighted term, the optimizer can temporarily accept physically impossible states and struggle to recover. Hard constraints are enforced as immediate rejection — any candidate that violates them is discarded before evaluation. Soft objectives are only scored on feasible solutions. This keeps the energy landscape smooth and the optimizer focused on business trade-offs, not physics.

WHY PRIORITY-BASED PRODUCT SELECTION?

Newly added products need good placement most — spending iterations on stable products wastes compute

At high temperature, 60% of neighbor generation targets newly added products, 30% recently adjusted, 10% stable. As temperature cools the ratio flattens. This focuses exploration budget where it matters most and speeds convergence significantly compared to uniform random selection.

// Rejected: Uniform random product selection (wastes iterations on stable products)
WHY RAY FOR PARALLELISM?

Each planogram optimization is independent — actor-based parallelism maps perfectly

A single optimization run takes 40–75 seconds. Processing 100+ planograms sequentially is not viable. Ray's actor model lets each planogram run in an isolated process with no shared state — no locks, no coordination overhead. 4–16 actors run concurrently depending on available CPU, giving near-linear throughput scaling.

// Rejected: asyncio only (CPU-bound work blocks the event loop), threading (GIL limits true parallelism for compute-heavy tasks)
WHY WEBSOCKET FOR STATE MUTATIONS?

The backend owns spatial validity — mutations must be validated server-side, not locally

Each move operation (position change, facing adjustment) is sent to the backend via WebSocket and validated there. The server recalculates available shelf space and returns the updated state. This ensures the optimizer always works with physically valid state and produces a full audit trail of every mutation — a compliance requirement for retail category management systems.
