A real-time, 3-stage recommendation pipeline for conference networking — semantic retrieval, learning-to-rank reranking, and feature-based filtering. Built for 1000+ concurrent users at sub-200ms latency.

Vector Search LambdaMART HNSW Change Data Capture AWS Bedrock FastAPI AWS ECS Stella400M
Problem

Keyword matching doesn't find the right people.

At large professional events, attendees with genuinely aligned backgrounds and interests fail to connect — because traditional filters operate on job titles and industry tags, not professional identity.

The system needed to understand what someone actually knows and has done, and surface the most relevant connections in real time as profiles updated throughout the event.

Architecture

3-Stage Pipeline

STAGE 01

Semantic Retrieval

HNSW index on Stella400M embeddings (768-dim). Weighted ingestion: Q&A 50% · Experience 30% · Education 20%. Returns top-100 candidates.

<50ms
STAGE 02

LambdaMART Reranking

Learning-to-rank over 100 candidates. Signals: interaction history, profile completeness, engagement patterns. Gradient boosting optimized for NDCG.

<100ms
STAGE 03

Feature Filtering

Hard filters: availability, geography, seniority, industry. Event-context scoped. Runs last to avoid polluting the ranking signal.

<20ms
REAL-TIME

Change Data Capture

MongoDB Change Streams on 6 collections. Profile update triggers async embedding regeneration — main request path never blocked.

<2s regen
// Data flow Profile update → Change Stream → Async task → Stella400M embed → HNSW index GET /recommend → HNSW cosine similarity top-100 <50ms → LambdaMART reranking <100ms → Feature filter <20ms → Response <200ms total
Design Decisions

Why these choices, not others.

WHY HNSW?

O(log n) retrieval at scale — exact search doesn't hold up at 1000+ concurrent users

Flat exact search would require scanning all profiles for every request. HNSW gives approximate nearest-neighbour search with tunable recall — we traded a small accuracy loss for the latency guarantee we needed. Index updates on profile change are incremental, so real-time CDC stays fast.

// Rejected: Flat FAISS index (no concurrent update support), Pinecone (cost at this scale)
WHY LAMBDAMART OVER A CROSS-ENCODER?

We needed to rerank on behavioural signals, not just semantic similarity

Cross-encoders are strong on semantic relevance but blind to engagement history, profile completeness, and interaction patterns. LambdaMART is a gradient-boosted L2R model that directly optimises NDCG — it learns from past connection outcomes, not just text similarity. Faster than a cross-encoder at inference too.

// Rejected: Cross-encoder reranker (ignores behavioural signals), simple score fusion (no learning)
WHY STELLA400M?

Better semantic compression of professional identity than generic models

Tested against OpenAI text-embedding-ada-002 and BGE-M3. Stella400M had better recall on professional profile similarity tasks — particularly on Q&A style text, which carries the most signal in this domain. Smaller than BGE-M3, so embedding generation stays under 2s per profile.

// Rejected: ada-002 (weaker on professional Q&A), BGE-M3 (slower regen cycle)
WHY WEIGHTED FEATURES (Q&A 50% / EXP 30% / EDU 20%)?

Not all profile fields carry equal networking signal

Q&A responses are free-form and reveal actual expertise and interests — highest signal. Experience descriptions carry role and domain context. Education is a weak signal for professional alignment at this event type. Weights were determined empirically from A/B testing on pilot events before production rollout.

WHY FILTER LAST, NOT FIRST?

Filtering before ranking contaminates the candidate pool for the reranker

Applying hard filters (geography, seniority) before LambdaMART reranking removes candidates the model might rank highly for other reasons. Running filters as the final stage means the reranker always sees a semantically rich candidate pool, and we only cut at the end for hard constraints.
