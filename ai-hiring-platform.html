AI Hiring
Platform

A production LangGraph multi-agent hiring system on Azure Container Apps — three specialised agents handling resume screening, interview analysis, and LLM-as-judge candidate ranking, with RBAC across 6 roles, ProctorAgent transcript NLP, and full LLMOps observability via Langfuse and MLflow.

Problem

Hiring decisions at scale are inconsistent — and the inconsistency is invisible.

Human hiring processes at volume suffer from a structural problem: different reviewers apply different standards to the same rubric. A resume that passes one screener fails another. An interview that one panellist scores highly gets downgraded by a second. The variance is real but unaudited — organisations have no signal on whether their evaluations are consistent, let alone calibrated to actual job performance. A platform that automates screening without solving consistency just scales the noise.

The second problem was process integration. Hiring involves multiple stakeholders — recruiters, hiring managers, interviewers, HR admins, executives — each needing different views and permissions. A system without granular access control forces a choice between exposing everything to everyone or building bespoke views per role. The platform needed to model the real org structure, not flatten it.

Architecture

Three Agents · One Graph · Full Observability

AGENT 01

Resume Screener

Parses and evaluates resumes against job description criteria. Structured output: skill match scores, experience gap flags, shortlist decision with reasoning. Feeds candidate state into the LangGraph shared context.

AGENT 02

Interview Analyser

ProctorAgent monitors live interview sessions. ScriptAnalyser runs NLP over transcripts post-interview: competency scoring, response quality signals, red flag detection. Both outputs written to MySQL.

AGENT 03

Candidate Ranker

LLM-as-judge: receives full candidate dossier (resume score + interview analysis) and ranks against the cohort using a structured rubric. Produces ranked shortlist with per-criterion justification.

Auditable reasoning
LLMOPS

Langfuse + MLflow

Langfuse traces every agent call — prompt version, token usage, latency, output. MLflow tracks evaluation experiments. RBAC (6 roles) enforced at API layer via Azure AD SSO + JWT.

Design Decisions

Why these choices, not others.

WHY LANGGRAPH OVER AUTOGEN?

Hiring has deterministic process stages — the agent graph must be explicit, not emergent

AutoGen's agent communication model is conversational: agents negotiate with each other to decide what happens next. This is appropriate for open-ended research tasks. A hiring pipeline is not open-ended — Resume Screener must complete before Interview Analyser runs, and Candidate Ranker only fires when both upstream agents have written their outputs. LangGraph's explicit state machine encodes these dependencies as graph edges, making the process auditable and reproducible. When a ranking decision needs to be explained to a candidate or an HR auditor, you can trace exactly which agent produced which output at which step — something emergent agent communication cannot guarantee.

// Rejected: AutoGen (emergent agent communication, non-deterministic execution order, hard to audit), CrewAI (role abstractions too high-level to model fine-grained hiring stage dependencies)
WHY LLM-AS-JUDGE FOR CANDIDATE RANKING?

Rule-based scoring cannot weigh qualitative evidence — LLM-as-judge can, and it produces auditable justifications

Traditional ATS systems score candidates on keyword density and field-matching rules. These are gameable and brittle — a resume optimised for ATS keywords scores higher than a genuinely stronger candidate who writes naturally. LLM-as-judge receives the full candidate dossier: resume analysis, competency scores from transcript NLP, interview quality signals. It applies a structured rubric and produces a ranking with per-criterion reasoning for every decision. This means every placement in the ranked shortlist is explainable — the hiring manager sees not just who ranked first but why, with citations back to specific evidence from the candidate's record.

// Rejected: rule-based weighted scoring (gameable, no qualitative reasoning, cannot weigh contradictory signals), embedding similarity ranking (produces a score, not a justification)
WHY THREE SEPARATE AGENTS INSTEAD OF ONE MONOLITHIC LLM?

Each hiring stage has different context requirements, failure modes, and prompt disciplines — a monolith conflates all three

A single LLM handling resume screening, interview analysis, and ranking simultaneously receives a context window containing all three stages of information. This creates prompt interference: instructions for ranking bleed into screening behaviour, and the model has no clear boundary between evaluation phases. Separate agents each have a focused system prompt, a defined input schema, and a single output responsibility. When the Interview Analyser produces unexpected output, the failure is isolated — it doesn't corrupt the Resume Screener's prior work. Debugging, prompt iteration, and evaluation can happen per-agent without touching the others.

// Rejected: single LLM with all hiring context (prompt interference between stages, large context cost, no isolation for debugging), two agents merging screening and analysis (interview evidence contaminating resume evaluation)
WHY PROCTORAGENT + SCRIPTANALYSER AS SEPARATE NLP COMPONENTS?

Live monitoring and post-hoc analysis are different tasks with different latency requirements

ProctorAgent operates during the interview: it flags anomalies in real time — candidate reading from notes, unusual pause patterns, connection drops — and writes session metadata continuously. ScriptAnalyser runs post-interview on the full transcript: it has access to the complete text and can run more expensive NLP (competency tagging, response quality scoring, coherence analysis) without a real-time latency constraint. Merging them would force real-time processing onto the heavier NLP workload or delay live session monitoring until post-processing — neither is acceptable.

// Rejected: single transcript analyser (cannot monitor live sessions), batch analysis of everything post-interview (loses real-time proctoring signal entirely)
WHY LANGFUSE + MLFLOW TOGETHER?

Langfuse is for production tracing — MLflow is for evaluation experiments. They solve different problems.

Langfuse instruments every live agent call in production: it captures the exact prompt sent, the model's response, token counts, latency, and prompt version. This is the operational record — what the system actually did, not what it was tested to do. MLflow is used for offline evaluation: comparing prompt versions against a labelled evaluation set, tracking which rubric formulation produces better ranking accuracy. Using only Langfuse means having production traces but no offline evaluation discipline. Using only MLflow means having experiment tracking but no visibility into how the deployed system is behaving on real traffic.

// Rejected: Langfuse only (no offline eval tracking, no prompt version comparison against ground truth), no observability (LLM outputs silently degrade with prompt drift, no signal until a hiring manager complains)
WHY RBAC WITH 6 ROLES VIA AZURE AD SSO?

Hiring data is sensitive and access patterns are structurally different per stakeholder — flattening roles creates security and UX problems simultaneously

A candidate should see only their own application status. An interviewer should see the candidate's resume and interview slot but not other candidates' scores. A recruiter needs pipeline-wide visibility without access to compensation data. A hiring manager needs team-specific data. HR admin needs audit logs. Superadmin manages the platform. Flattening to three roles (candidate/reviewer/admin) either over-privileges interviewers or under-privileges recruiters — both are wrong. Azure AD SSO means role assignment is managed in the client's existing identity provider, not a custom user database, and JWT tokens carry role claims that RBAC middleware enforces at every API route.

// Rejected: three-tier role model (too coarse, over-privileges interviewers), custom auth system (reinvents identity management the client already has in Azure AD)
