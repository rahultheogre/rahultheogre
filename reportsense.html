ReportSense
AI

A hybrid RAG pipeline for Tableau BI report discovery and natural language recommendation — BGE-M3 dense retrieval fused with BM25 via RRF, FlashRank reranking, and LLM query expansion. Reduced LLM operating cost by ~60%.

RAG BGE-M3 BM25 RRF Fusion FlashRank Query Expansion Tableau FastAPI
Problem

Analysts couldn't find the right Tableau report — and couldn't describe what they were looking for in keywords.

Large BI environments accumulate hundreds of Tableau dashboards and workbooks over years. Finding the right report required knowing its exact name or navigating folder structures built by different teams with inconsistent naming conventions. Analysts queried natural language — "show me regional sales by product category for last quarter" — but the existing search returned results based on title keywords only, missing semantically equivalent reports named differently.

The second problem was cost. An early version passed entire report metadata catalogs into an LLM context window for each query, hitting token overflow errors on large environments and driving high per-query LLM spend. The system needed smarter retrieval that reduced what the LLM had to read — not just what it could find.

Architecture

Query Expansion → Hybrid Retrieval → Rerank → Generate

STAGE 01

LLM Query Expansion

User query rewritten into multiple reformulations before retrieval. Captures synonyms, alternative phrasings, and implicit intent — increases recall before the retrieval stage runs.

STAGE 02

Hybrid Retrieval

BGE-M3 dense retrieval (semantic) + BM25 lexical retrieval run in parallel. RRF fusion: dense weight 0.65 / BM25 weight 0.35, k=60. Merged candidate set passed forward.

STAGE 03

FlashRank Reranking

ms-marco-MiniLM-L-12-v2 reranker scores merged candidates jointly against the query. Threshold 0.65 — below this, candidates are dropped before LLM sees them.

80K token hard limit
STAGE 04

LLM JSON Spec Generation

Only reranked, threshold-passing chunks reach the LLM. Output: structured JSON spec — report ID, relevance rationale, recommended filters. Round-robin chunk distribution prevents token overflow.

~60% cost reduction

Design Decisions

Why these choices, not others.

WHY QUERY EXPANSION BEFORE RETRIEVAL?

A single user query is often underspecified — expansion recovers recall before the retrieval bottleneck

Users phrase BI queries colloquially: "profitability by region" might match a report titled "P&L Geographic Breakdown Q3." Without expansion, the dense retriever produces an embedding of the original phrasing and may miss reports whose metadata uses different but equivalent terminology. Running LLM-driven query expansion before retrieval generates semantically equivalent reformulations — the retriever then runs against a richer query signal, not just the user's first draft. The cost of one small LLM call at query expansion is recovered many times over by reducing the number of irrelevant chunks the expensive reranker has to process.

// Rejected: retrieval on raw query only (misses terminology mismatch between user language and report metadata), post-retrieval query rewriting (too late — bad first-stage recall can't be fixed by rewriting)
WHY HYBRID RETRIEVAL OVER PURE DENSE?

BI report names and filter fields are exact strings — embedding space cannot reliably distinguish them

Tableau report metadata contains exact identifiers: workbook names, data source names, calculated field names, filter parameter strings. These are not paraphraseable — "Sales_Region_FY24" and "Revenue_Geography_FY24" may be semantically similar but are distinct reports. Dense retrieval treats them as nearly identical. BM25 treats character-level differences as significant, which is exactly what you want for identifier-heavy metadata. The hybrid merge means semantic intent and exact-match precision both contribute to the candidate pool.

// Rejected: pure BGE-M3 dense retrieval (poor discrimination on exact report identifiers and filter strings), pure BM25 (no handling of paraphrase or natural language queries)
WHY RRF OVER WEIGHTED SCORE FUSION?

Dense and BM25 scores live in incompatible spaces — rank-based fusion is scale-invariant

Dense retrieval returns cosine similarity scores in [−1, 1]. BM25 returns TF-IDF-derived scores with no fixed upper bound, varying with document length and corpus statistics. Directly multiplying these by weights and summing them produces a number that means nothing — a BM25 score of 12 and a cosine similarity of 0.87 are not commensurable. Reciprocal Rank Fusion operates on rank positions, which are scale-invariant — it doesn't matter what unit the underlying score is in. The k=60 smoothing parameter prevents top-ranked items from dominating when one retriever has an outlier score.

// Rejected: weighted score sum (scores not commensurable across retrieval methods), simple score normalisation then sum (normalisation is corpus-dependent, degrades under distribution shift)
WHY FLASHRANK OVER A FULL CROSS-ENCODER?

Cross-encoders are accurate but slow — FlashRank delivers equivalent quality at a fraction of the latency

A standard cross-encoder like ms-marco-MiniLM runs a full transformer forward pass for every query-document pair. At our candidate set sizes post-RRF, this was the latency bottleneck. FlashRank implements the same ms-marco-MiniLM-L-12-v2 model with optimised inference — it uses late interaction scoring to avoid redundant computation across candidates that share query tokens. The 0.65 threshold was calibrated empirically: it drops candidates the LLM would score low anyway, further reducing the token payload that reaches the generation stage.

// Rejected: standard cross-encoder (correct but slow at this candidate volume), no reranking (RRF alone left too much noise in the top-k for reliable LLM generation)
WHY PARSER FACTORY + CENTRALISED PROMPT MANAGEMENT?

Multi-source support and prompt drift are engineering problems, not ML problems — solve them structurally

The client operated both Tableau and PowerBI environments. Without a parser abstraction, adding PowerBI support would mean forking the retrieval logic. A parser factory with a shared normalised schema means the entire pipeline downstream — retrieval, reranking, generation — is source-agnostic. Centralised prompt management solved a different problem: with three engineers on the team, prompts were being edited in multiple places and diverging silently. A single prompt registry with versioned templates made prompt changes auditable and prevented the LLM's output format from drifting between environments.

// Rejected: separate retrieval pipelines per BI source (code duplication, inconsistent behaviour), prompts embedded in route handlers (unauditable, three engineers editing the same strings in different files)
